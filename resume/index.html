How I build with LLMs

Most of my AI work is at the application layer — turning models into features people actually use. Not model training or on-device inference optimization. Concretely:

I chose not to use a vector DB — and that was the right call

The default reflex is "RAG = embeddings + vector store." My domain data had a well-defined schema, so key-based lookup in MongoDB was more accurate, faster, and cheaper than embedding search — semantic similarity would have retrieved plausible-but-wrong records. So it's DB-backed structured RAG: multiple context sources (user profile, life context, emotional trend, prior sessions, cross-references) composed into one prompt per request, not top-k chunks glued together.

mongodb · structured rag

Context window as a budget

Six consultation types, each pulling only the context that type actually needs — cutting API cost without cutting quality. Response params are adapted per question: complexity is classified (simple / moderate / deep) and max_tokens (350–800) and temperature (0.7–0.85) are set accordingly, instead of one fixed config for every call.

token budget · dynamic params

Evaluation with actual statistics

I built a 6-dimension LLM-as-Judge harness as a quality gate on prompt and model changes:

  • Self-bias avoided — GPT-4o judges, GPT-4o-mini generates (never the same model grading itself)
  • Judge reliability measured — Cohen's quadratic weighted kappa, so I know the judge is trustworthy
  • Regressions caught statistically — Welch's t-test with Bonferroni correction, not eyeballing a few samples
  • CoT + score calibration + few-shot; judge prompts versioned and hashed
llm-as-judge · qwk · t-test

Safety designed in, not bolted on

A counseling app receives real distress. Crisis detection runs in real time and grades into two levels — watch vs. urgent — with urgent cases routed out of the generative path and into real crisis hotlines. On the analytics side, a 5-stage filter means only positive, anonymized sessions ever feed service-level learning; negative or personal data is structurally blocked from touching another user.

2-tier crisis routing · privacy

A system that learns from each session

Facts are extracted from conversation and written back to the user's context immediately. Life changes are stored as history rather than overwrites, so "what recently changed" is answerable. Emotion scores are tracked over time to classify a trend as improving / stable / declining, and what a user revealed in one consultation type is available to the others.

stateful · cross-reference
